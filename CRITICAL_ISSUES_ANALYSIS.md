## Executive Summary

The system has **4 critical issues** stemming from **architectural inconsistencies** between:
- **Monolithic Pattern** (new): Uses `conversation_history` as source of truth
- **Legacy Pattern** (old): Uses `pending_questions` and separate storage

This causes cascading failures in question generation, answer processing, suggestions, and frontend state management.

### Root Cause Analysis

**Location 1**: `orchestrator.py:1797-1810`
```python
pending_questions = getattr(project, "pending_questions", []) or []
phase = getattr(project, "phase", "discovery")
previously_asked = self._extract_previously_asked_questions(pending_questions, phase)
```

**Location 2**: `orchestrator.py:3044-3083` (Deduplication logic)

```python
if conversation_history:
    for msg in conversation_history:

            msg.get("phase") == phase and
            msg.get("content")):
```

### The Problem

**MONOLITHIC VS LEGACY PATTERN MISMATCH:**

1. **Question generation context** (line 1797) gets previously asked questions from `pending_questions`
   - This is the OLD pattern
   - `pending_questions` is NOT being populated in monolithic mode

2. **Question deduplication** (line 3044) extracts from `conversation_history`
   - This is the NEW monolithic pattern
   - But question is already generated by then

3. **Result**:
   - Counselor is asked to generate question
   - Passes previously_asked list (which might be empty or outdated)
   - Counselor sees an unanswered question in its internal state
   - Returns "Returning existing unanswered question for Themis" instead of generating new one
   - Question gets cached but NOT stored in conversation_history
   - Next request: Deduplication finds the same question still unanswered
   - LOOP: Keeps returning same question

### The Evidence Chain

### Why It's Happening

**Missing Link**: There's NO code that:
1. Checks if an unanswered question exists in `conversation_history`
2. Instructs counselor to return it instead of generating a new one
3. Handles the case where counselor deliberately returns unanswered question

The counselor has internal logic to return existing unanswered questions, but the orchestrator doesn't know how to handle that response.

## ISSUE #3: Suggestions Generation - ROOT CAUSE FOUND ✓

### Symptom
Suggestions endpoint returns generic phase-aware suggestions instead of context-aware suggestions:
```
Suggested answer starters:

2. Who are your target users?
```

And error in logs:

### Log Evidence
```

[ANSWER_PROCESSING] Step 1 Result: Extracted 0 total specs
[GET] /projects/proj_3d4beb3fcc1e/chat/suggestions
[WARNING] Suggestion generation failed: Question  not found
[Response] suggestions: [generic list...]
```

### Root Cause Analysis

**Location 1**: `orchestrator.py:2082-2090` - Question finder
```python
def _find_question(self, project, question_id: str) -> Optional[Dict]:
    """Find a question by ID in pending_questions."""
    pending_questions = getattr(project, "pending_questions", []) or []
    for q in pending_questions:
        if q.get("id") == question_id:
            return q
    return None  # ← Returns None because pending_questions is empty
```

**Location 2**: `orchestrator.py:2127-2135` - Suggestion generation
```python
question = self._find_question(project, question_id)
if not question:
    return {"status": "error", "message": f"Question {question_id} not found"}
```

**Location 3**: `projects_chat.py:1710-1788` - Endpoint
```python
current_question = None
if project.pending_questions:
    unanswered = [q for q in project.pending_questions if q.get("status") == "unanswered"]
    if unanswered:
        current_question = unanswered[-1].get("question")
```

### The Problem

**DUAL DATA STRUCTURE MISMATCH:**

1. Questions are stored in `conversation_history` (monolithic pattern)
2. Suggestion code looks in `pending_questions` (legacy pattern)
3. `pending_questions` is empty or not populated
4. Question lookup fails
5. Falls back to generic suggestions

### What Should Happen

**Monolithic Pattern**:
- Questions: `project.conversation_history` (messages with `type="assistant"`)
- Specs: `project.goals`, `project.requirements`, etc.

**Legacy Pattern**:
- Questions: `project.pending_questions` (separate tracking)
- Specs: `project.context.goals`, etc.

The system is mixing both patterns, causing lookups to fail.
